# webpack基本配置介绍

# 6大核心概念

# 1. entry 入口

Webpack 行构建的第一步将 Entry 开始，可抽象成输入。

# 2. output 输出结果

在 Webpack 经过 系列处理并得出最终想要的代码后输出 结果。

# 3. plugin 拓展插件

在 Webpack 构建流程中的特定时机注入扩展逻辑，来改变构建结 果或做我们想要的事情。

# 4. loader 模块转换器

用于将模块的原内容按照需求转换成新内容。

# 5. chunk 代码块

一个chunk由多个模块组成，用于代码分割和合并

# 6. module 模块

webpack，一切皆模块，一个文件一个模块

# 运行流程

Webpack 在启动后会从 Entry配置的 Module 开始，递归解析 Entry 依赖的所有 Module,每找到一个 Module ，就会根据配置的 Loader 去找出对应的转换规则，对 Module 进行转换后，再解析出当前 Module 依赖的 Module 这些模块会以 Entry为单位进行分组，Entry及其所有依赖的Module被分到一个组也就是 Chunk 。最后， Webpack 将所有 Chunk 转换成文件输出。在整个流程中， Webpack 会在恰当的时机执行 Plugin定义的逻辑。

# 配置

# Entry ：配置模块的入口．

可以接受的值 单个入口：str、arr； 多个入口 obj 也可以接受promise 动态注册入口

# Output：配置如何输出最终想要的代码

  1. ...
  2. filename 可以选择内置模板[id],可选id,name,hash,chunkhash
  3. path 文件输出的目录路径
  4. publicPath 发布线上的url，添加cdn，默认'/'
  5. crossOriginLoading 默认:anonymous 不带用户cookies,use-credentials 带上用户cookies
  6. library 导出库的变量名.
  7. libraryTarget 配置以何种方式导出库 var(默认)，commonjs，commonjs2，this，window，global，umd，umd2，amd，jsonp
  8. pathinfo 是否包含有用的文件路径信息到生成的代码里 ，为 boolean 类型
  9. chunkFilename 附加 Chunk 的文件名称

# Module ：配置处理模块的规则

  1. noParse 正则匹配名字不解析
  2. rules {test,include,exclude，parse}

# Plugins ：配置扩展插件

plugins 配置项接收一个数组，数组里的每一项都是一个要使用 Plugin 的实例， Plugin 需要的参数通过构造函数传入

# DevServer：配置 DevServer

  1. inline 是否插入socket，否用iframe打开，默认true
  2. historyApiFallback boolean or {rewrites: [{form,to}]}
  3. contentBase path.join( dirname,'public');
  4. headers 写入请求头
  5. host 0.0.0.0 提供局域网访问
  6. port 端口
  7. allowedHosts ['abc.com','asd.com']，允许dns的白名单
  8. disableHostCheck 不必host的检查，允许外网配置host
  9. https true(随机生成证书) or {证书路径}
  10. clientLogLevel 配置客户端的日志等级,可取如下值: none error warning info 。
  11. compress 是否启用gzip
  12. open 是否自动用默认浏览器打开项目 /
  13. openPage 指定open打开的url

# resolve

  1. alias 别名{key:path/to/file.xx}
  2. mainFields [jsnext:main,browser,main]从模块里设置主要入口
  3. extensions ['.js','ts']自动补全后缀名
  4. modules webpack在什么路径下找依赖模块默认['node_modules']
  5. descriptionFiles 依赖包描述文件，默认['package.json']
  6. enforceExtension 是否必须带后缀名 默认false
  7. enforceModuleExtension 是否必须带后缀名（ 只对 node_modules 下的模块生效） 默认false
  8. symlinks: true , 是否跟随文件的软链接去搜寻模块的路径

# target 输入运行在不同环境文件

target 配置项可以让 Webpack 构建出针对不同运行环境的代码 可选值 web、node、async-node、webworker、electron-main、electron-renderer

# devtool 配置 Webpack 如何生成 Source Map

默认值是 false，可选值

# watch & watchOptions 是否监听文件修改 & 可选参数

watch boolean 默认false

{
    ignored: /node modules/,
    ／／监听到变化后会等 300ms 再去执行动作，防止文件更新太快导致重新编译频率太高
    ／／默认为 300ms
    aggregateTimeout: 300,
    ／／判断文件是否发生变化是通过不停地询问系统指定文件有没有变化实现的
    ／／默认每秒询 1000
    poll: 1000
}

# Externals 不打包的库{key: 库的全局变量}

# Resolveloader 用来告诉 Webpack 如何去寻找Loader,该配置项常用于加载本地的 Loader

  1. modules ['node_modules']
  2. extensions 自动补全后缀名
  3. mainFelds 指明入口文件位置的字段

# stats // 控制台输出日志控制

assets : true, colors : true, errors: true, errorDetails: true, hash : true ,

# profile // 是否捕捉 Webpack 构建的性能信息，用于分析是什么原因导致构建性能不佳

# cache // 是否启用缓存来提升构建速度

# 总结：通常我们可用如下经验去判断如何配置 Webpack

若想让源文件加入构建流程中被 Webpack 控制，则配置 entry; 若想自定义输出文件的位置和名称 ，则配置 output; 若想自定义寻找依赖模块时 策略 ，则配置 resolve; 若想自定义解析和转换文件的策略， 则配置 module ，通常是配置 module. rules 里的 Loader; 若其他大部分需求可能通过 Plugin 去实现 ，则配置 plugin
